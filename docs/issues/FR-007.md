# [FR-007] 120fps Object Pooling

## Labels
`Future`, `optimization`, `performance`

## User Story
As a developer, I want to recycle bullet and enemy objects so that performance never dips below 50fps, even in extreme bullet hell scenarios with hundreds of active entities.

## Proposed Solution

### Overview
The "heavy" feeling in the current game is largely caused by the browser's Garbage Collector (GC) cleaning up thousands of discarded bullet and particle objects. This overhaul replaces `new` object creation with a static "Recycle" system (Object Pooling), ensuring the game's memory footprint stays flat.

### Implementation Flow
1. Pre-allocate large "Dead" arrays for all high-frequency entities (`Laser`, `EnemyBullet`, `Particles`) during the loading screen.
2. Replace all `spawn` logic with a `get()` call that pulls an existing object from the "Dead" array.
3. When an entity dies, "Recycle" it by pushing it back into the "Dead" array instead of letting it be deleted.

### Technical Approach
```javascript
// Pre-allocate objects at startup to avoid mid-game "allocations"
const laserPool = Array.from({length: 100}, () => new LaserBullet());
```

### Key Considerations
- **GC-Free**: The goal is zero `new` keywords after the game starts.
- **Stateless Recycle**: Ensure objects are completely reset before being reused to prevent "ghost" data.
- **Fixed Size**: Use fixed-size arrays for maximum speed.

## Acceptance Criteria
- [ ] No `new` keywords are used in the core game loop for bullets or particles.
- [ ] The game maintains a locked frame rate (50-90fps) during Adrenaline Surges.
- [ ] Memory profiling shows a flat line (no sawtooth GC patterns).
